PR #7026: avoid DB DS-list refresh on stale-dataset status updates

This page is an onboarding guide for reviewing this PR with zero prior context.

Author: mihir20 Base ← Head: mihir/ai-preview ← mihir/pr-preview-test Main area: jobsdb Other area: .github/workflows/ai-preview.yaml

TL;DR

The core fix changes how jobsdb.UpdateJobStatusInTx recovers from ErrStaleDsList. Instead of re-reading dataset metadata from PostgreSQL inside the retry loop, it now reuses the already-maintained in-memory dataset snapshot under a read lock. This removes a DB-connection dependency from a hot retry path that can deadlock under high updater concurrency.

The PR also adds a heavy compaction benchmark scenario to measure drain-time under three compaction modes, and updates the CI preview workflow to run Codex with a newer model plus sandbox bypass on GitHub-hosted runners.

Why This Change Exists

In jobsdb, datasets ("DS") can change shape during background compaction/migration. An in-flight status update may then target a dataset that has become stale, which surfaces as ErrStaleDsList. Retrying is expected behavior.

The old retry path in UpdateJobStatusInTx refreshed DS metadata from the database through the same transaction/pool context. Under saturation, status-updater goroutines can occupy all main pool connections; then no connection remains for refresh, so progress stalls. This is exactly the failure mode described in the PR description.

The repo already keeps an authoritative in-memory DS snapshot updated by maintenance loops (for example after compaction publish). This PR aligns retry behavior with that invariant: read current DS list from memory, not from DB, before retrying status updates.

What Changed (Grouped by Concern)

1) Status-update stale retry path (jobsdb/jobsdb.go)

  • UpdateJobStatusInTx now handles ErrStaleDsList by taking dsListLock read lock, snapshotting jd.dsList, and updating the external UpdateSafeTx snapshot via tx.setDSList(...).
  • The old path that called doRefreshDSRangeListWithDB(..., tx.SqlTx()) was removed from this retry branch.
  • internalUpdateJobStatusInTx now suppresses generic error logging for ErrStaleDsList (expected transient) and logs only non-stale failures.
  • New option WithTriggerAddNewDS was added to configure deterministic addNewDS triggers (used by new benchmark scenario).

2) New compaction benchmark scenario (jobsdb/bench/*)

  • Added scenario selector support in jobsdb/bench/bench.go for "compaction".
  • Added heavy benchmark runner in jobsdb/bench/scenario/compaction.go:
  • Seeds datasets * jobsPerDataset jobs with deterministic failure pattern.
  • Drains with one consumer goroutine per destination and repeated GetToProcess + UpdateJobStatus.
  • Compares three flag sets: baseline, nonBlockingCompletedDSDrop, and nonBlockingCompaction.
  • Supports optional dedicated maintenance connection pool via config.
  • bench_test.go adds TestBench/compaction, skipped unless RUN_COMPACTION_BENCH=1.

3) Migration/add-DS observability tweaks

  • Logging string in addNewDSLoop was prefixed for clarity.
  • migration.go adds explicit lock duration logs in both migrate and compact paths.

4) Workflow adjustments (.github/workflows/ai-preview.yaml)

  • Codex model name changed from gpt-5-codex to gpt-5.3-codex.
  • Codex invocation changed from --sandbox workspace-write to --dangerously-bypass-approvals-and-sandbox with rationale comments about GitHub runner constraints.

Architecture / Flow Diagrams

The important connection is between stale-retry and the in-memory DS cache; DB refresh is removed from this retry path.

Sequence: stale update retry

Focus on the retry branch: stale error triggers a memory snapshot refresh instead of DB I/O.

Before / After (retry refresh source)

Before: on stale DS, retry did doRefreshDSRangeListWithDB(..., tx.SqlTx()) while holding write lock logic, requiring DB connection availability inside retry path.

After: on stale DS, retry acquires dsListLock read lock, snapshots in-memory DS metadata, and updates UpdateSafeTx snapshot. No DB call in that branch.

File-by-File Walkthrough

jobsdb/jobsdb.go - core behavior change

This is the critical fix. The stale retry no longer refreshes from DB in this call path.

Key hunk: stale retry refresh source

...
if refreshErr := func() error {
    if lock := jd.dsListLock.RTryLockWithCtx(ctx); !lock {
        return fmt.Errorf("acquiring read lock for refreshing ds list in update job status: %w", ctx.Err())
    }
    dsList, dsRangeList := jd.dsList.snapshot()
    tx.setDSList(dsList, dsRangeList)
    jd.dsListLock.RUnlock()
    return nil
}(); refreshErr != nil {
    return refreshErr
}

Cross-File Context (What Else Matters)

  • Where stale comes from: updateJobStatusDSInTx converts readonly-table PG error (RS001) into ErrStaleDsList, which is the retry trigger.
  • Who maintains DS cache: flows like compaction/migration call doRefreshDSRangeList after publish, and addNewDSLoop refreshes DS list after create/append events.
  • Read-path parallel: inStoreSafeCtx already had stale retry logic that may refresh DS metadata under lock. This PR makes update-path retry more lightweight than before by avoiding DB in that retry branch.
  • Benchmark realism choices: compaction scenario pins dsLimit, uses destination filters, and treats empty reads as transient because jobs can be hidden behind DS window until compaction advances.
  • New option blast radius: WithTriggerAddNewDS is a narrow API addition currently used by compaction benchmark; existing tests often set TriggerAddNewDS directly on handle structs.

How To Review This PR

  1. Read jobsdb/jobsdb.go first: UpdateJobStatusInTx retry loop and internalUpdateJobStatusInTx logging guard.
  2. Validate stale error generation source in updateJobStatusDSInTx and ensure retry still properly rebinds DS/range snapshot before reattempt.
  3. Read jobsdb/jobsdb_test.go test update to confirm this invariant: retry assumes DS cache has already been refreshed by maintenance path.
  4. Read new benchmark scenario (jobsdb/bench/scenario/compaction.go) with focus on determinism, concurrency safety, and whether it models intended production contention.
  5. Skim workflow diff last, as separate operational concern from jobsdb behavior.

What to scrutinize closely

  • Failure semantics when RTryLockWithCtx fails: error now depends on context cancellation/deadline rather than DB failure.
  • Assumption that in-memory DS list is always refreshed before stale retry needs it; test now mirrors this explicitly.
  • Potential log-volume changes from new migration/add-DS informational logs in high-throughput environments.
  • Whether benchmark defaults/labels match comments (there is wording drift in comments around data volume).

Risks, Edge Cases & Things To Double-Check

high

Invariant dependency: stale retry now trusts in-memory DS metadata freshness. If a future code path triggers stale status updates without publishing DS refresh first, retries may continue targeting wrong ranges.

medium

Context-sensitive lock acquisition: retry now fails via context-expired lock acquire, so timeout values on callers become more important for recovery behavior under lock pressure.

medium

Benchmark is heavy and opt-in: no default CI guard covers this scenario. Regressions in the compaction benchmark path may go unnoticed unless teams run RUN_COMPACTION_BENCH=1.

low

Operational security posture change: preview workflow now explicitly bypasses Codex sandbox on runner. Rationale is documented, but this is still a policy-sensitive choice for repo maintainers.

low

Logging noise drift: additional lock-duration logs in migration paths may increase log volume during compaction-heavy workloads.
