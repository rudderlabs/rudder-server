TL;DR

This PR changes the stale-dataset retry path in jobsdb.UpdateJobStatusInTx: instead of re-reading dataset ranges from PostgreSQL, it now reuses the in-memory DS snapshot under a read lock. The goal is to avoid requiring a free DB connection during retry, which can deadlock/stall under high concurrency.

It also adds a heavy jobsdb/bench compaction benchmark scenario to reproduce/compare compaction strategies, plus small observability log adjustments. Separately, the PR updates the preview workflow model name and changes Codex invocation to bypass inner sandboxing on GitHub-hosted runners.

Why This Change Exists

Problem: status updates can race with compaction/migration that swaps dataset layout. If status writes land on a table made read-only by compaction trigger, code returns ErrStaleDsList and retries with refreshed dataset mapping.

Previous retry behavior: refresh mapping by querying DB (doRefreshDSRangeListWithDB(..., tx.SqlTx())). Under load, if updater goroutines consume the pool, there may be no free connection for that refresh call, so retry path blocks.

New approach: trust the in-memory dsList cache that compaction already updates after commit; acquire dsListLock read lock and copy snapshot into the update transaction context.

Domain note: “DS” means dataset shard pair (jobs_* and job_status_* tables). Status updates route by an in-memory range list; stale routing is expected during compaction and recovered by retry.

What Changed (Grouped By Concern)

1) Core behavior change in status-update stale retry

  • jobsdb/jobsdb.go: UpdateJobStatusInTx retry path no longer runs DB refresh; now does RTryLockWithCtx + dsList.snapshot() + tx.setDSList(...).
  • jobsdb/jobsdb.go: suppresses redundant info logging when the internal update error is exactly ErrStaleDsList (already handled by retry warning log).
  • jobsdb/jobsdb_test.go: existing stale-retry external-tx test now explicitly refreshes in-memory DS list after replacement setup to match new behavior.

File-by-File Walkthrough

jobsdb/jobsdb.go (core retry logic + option + log filtering)

Main functional change: stale-ds retry now reads already-published in-memory DS snapshot instead of running a DB refresh in the retry loop.

Retry refresh before vs after
// BEFORE (removed)
return jd.dsListLock.WithLockInCtx(ctx, func(l lock.LockToken) error {
  if err := jd.doRefreshDSRangeListWithDB(l, tx.SqlTx()); err != nil {
    return fmt.Errorf("refresh ds range list: %w", err)
  }
  dsList, dsRangeList := jd.dsList.snapshot()
  tx.setDSList(dsList, dsRangeList)
  return nil
})

// AFTER (new)
if lock := jd.dsListLock.RTryLockWithCtx(ctx); !lock {
  return fmt.Errorf("acquiring read lock for refreshing ds list in update job status: %w", ctx.Err())
}
dsList, dsRangeList := jd.dsList.snapshot()
tx.setDSList(dsList, dsRangeList)
jd.dsListLock.RUnlock()

Cross-File Context (Important For Correct Review)

How stale routing is generated

  • Compaction (jobsdb/migration.go#doCompactDS) adds a readonly trigger on source status tables before commit.
  • Status insert to old table then returns PG error code RS001 (mapped via pgErrorCodeTableReadonly), converted in doUpdateJobStatusInTx to ErrStaleDsList.
  • UpdateJobStatusInTx catches ErrStaleDsList, rolls back to savepoint, refreshes DS mapping, retries.

Reviewer focus: does this assumption hold everywhere — compaction must have already published fresh DS cache before stale retry occurs.

How To Review This PR

Suggested reading order

  1. jobsdb/jobsdb.go around UpdateJobStatusInTx and internalUpdateJobStatusInTx.
  2. jobsdb/jobsdb_test.go test TestUpdateJobStatusInExternalTxRetriesAfterReadonlyStatusTable.
  3. jobsdb/migration.go comments around readonly triggers in doCompactDS for retry assumptions.
  4. jobsdb/bench/scenario/compaction.go for benchmark methodology and determinism.
  5. .github/workflows/ai-preview.yaml for CI security/operational implications.

What to scrutinize

  • Locking correctness: read-lock snapshot + explicit unlock path has no deferred unlock; verify no early returns between lock acquisition and unlock in future edits.
  • Retry liveness: if compaction hasn’t published new dsList yet, repeated stale retries should still eventually converge.
  • Benchmark validity: scenario compares three strategies with shared fail pattern, but ensure goroutine capture behavior in seeding loop is safe (see risk below).
  • Workflow posture: bypassing Codex sandbox is justified in comments; reviewer should confirm organization policy permits this in CI.

Risks, Edge Cases, Double-Checks

High

Potential goroutine loop-capture bug in new compaction seeding

In jobsdb/bench/scenario/compaction.go, goroutines launched inside for start := 0; ... capture start directly. In Go, loop-variable capture in goroutines can cause workers to observe unintended values if not copied locally. This is benchmark-only code, but it can skew results or produce uneven seeding.

Double-check by verifying local copies are used per goroutine (e.g. batchStart := start).

Medium

Retry refresh error message may lose root cause context

RTryLockWithCtx returns false when context done. Current error wraps ctx.Err(); if caller passes active context but lock acquisition fails unexpectedly, message may include <nil>, reducing diagnosability.

Medium

Assumption coupling to compaction publication timing

The new retry path assumes DS cache is current because compaction refreshes it. Verify no alternative layout-changing paths leave DB newer than in-memory snapshot for extended periods.

Low

Log volume increase under busy migration loops

New lock-duration info logs in migration paths can increase log cardinality/noise if loop frequency is high.

Medium

CI hardening posture change in preview workflow

--dangerously-bypass-approvals-and-sandbox is a broader trust model than --sandbox workspace-write. Evaluate this against your organization’s CI threat model even though job runs on ephemeral GitHub-hosted runner.

Testing / Verification

What this PR adds or updates

  • Adapts existing stale-retry correctness test in jobsdb/jobsdb_test.go to match new in-memory refresh semantics.
  • Adds a new benchmark scenario test entry TestBench/compaction, intentionally skipped unless RUN_COMPACTION_BENCH=1.

What is not added

  • No new focused unit/integration test that directly asserts “stale retry does not perform DB refresh / does not require free pool connection”. Evidence is behavioral + benchmark narrative, not an explicit regression test for pool starvation.
  • No tests for workflow sandbox mode change (expected for CI YAML edits).

Reviewer quick checklist

  • Confirm stale retry path is safe with concurrent compaction and with cancellation contexts.
  • Confirm new benchmark is deterministic enough to compare strategy deltas across runs.
  • Confirm CI workflow security exception is intentional and acceptable.
