TL;DR

The `jobsdb` stale-dataset retry path no longer refreshes dataset metadata by querying Postgres inside `UpdateJobStatusInTx`. Instead, it takes a read lock and snapshots the already-published in-memory DS cache. This removes one DB connection dependency from a hot retry path and addresses a connection-pool starvation/stall scenario under heavy concurrent status updates.

The PR also adds a heavy benchmark scenario (`TestBench/compaction`) to compare compaction strategies under a controlled, large draining workload, plus unrelated but meaningful hardening in `.github/workflows/ai-preview.yaml` for Codex execution and `gh-pages` deployment hygiene.

Why This Change Exists

Domain translation: Jobs are partitioned into dataset pairs (`jobs_*`, `job_status_*`). Background maintenance (add-new-dataset, migration, compaction, and drop) changes that layout. Writers updating job statuses hold an update-safe view of the dataset list, but if layout changes mid-flight the write can hit `ErrStaleDsList` and must retry against newer layout.

Previously, that retry called `doRefreshDSRangeListWithDB` inside `UpdateJobStatusInTx`, which means it needed a DB connection while still in a pressure situation. The PR description’s failure mode is: many in-flight status updaters consume the entire main pool, then stale-list retries need an extra connection to refresh, but none are free, so progress can stall.

The key observation behind this patch: compaction/migration already republish DS list state to memory (`jd.dsList`) after layout changes. So stale-list retries only need a synchronized read of that in-memory cache, not another DB refresh.

What Changed (By Concern)

1. Retry behavior in `UpdateJobStatusInTx`

  • On `ErrStaleDsList`, code now `RTryLockWithCtx` on `dsListLock`, snapshots `jd.dsList`, and updates the transaction-local DS list/range (`tx.setDSList(...)`).
  • It no longer calls `doRefreshDSRangeListWithDB(..., tx.SqlTx())` in this retry path.
  • `internalUpdateJobStatusInTx` now suppresses info-level logging for expected stale-list errors to reduce noisy logs.

2. Benchmarking infrastructure

  • `jobsdb/bench/bench.go` adds scenario selector `compaction`.
  • New `jobsdb/bench/scenario/compaction.go` seeds many datasets, drains with one consumer per destination, precomputes deterministic failure patterns, and runs 3 compaction-flag scenarios under identical workloads.
  • `jobsdb/bench/bench_test.go` adds `TestBench/compaction`, gated by `RUN_COMPACTION_BENCH=1` because it is intentionally heavy.

3. JobsDB API/test adjustments

  • New option `WithTriggerAddNewDS` enables deterministic dataset rollover trigger control (used by the new benchmark).
  • Existing stale-retry unit test now explicitly refreshes the in-memory DS list before invoking retry, aligning test setup with new runtime assumption.
  • Minor logging tweaks in `addNewDSLoop` and migration/compaction lock-duration logs.

4. GitHub workflow hardening (`.github/workflows/ai-preview.yaml`)

  • Codex model name updated to `gpt-5.3-codex`.
  • Codex invocation now uses `--dangerously-bypass-approvals-and-sandbox` due to runner bubblewrap/network namespace limitations.
  • `gh-pages` initialization now creates a clean orphan branch without inherited tracked files; extra guard removes stray tracked files and rejects symlink content before push.

File-by-File Walkthrough

`jobsdb/jobsdb.go` (core behavior change)

In `UpdateJobStatusInTx`, stale-list retry now refreshes from in-memory snapshot instead of DB refresh. It still enforces synchronization by taking a read lock before snapshotting.

Key hunk (retry refresh strategy)
if refreshErr := func() error {
  if lock := jd.dsListLock.RTryLockWithCtx(ctx); !lock {
    return fmt.Errorf("acquiring read lock for refreshing ds list in update job status: %w", ctx.Err())
  }
  dsList, dsRangeList := jd.dsList.snapshot()
  tx.setDSList(dsList, dsRangeList)
  jd.dsListLock.RUnlock()
  return nil
}(); refreshErr != nil {
  return refreshErr
}

Secondary change: `internalUpdateJobStatusInTx` no longer logs `ErrStaleDsList` as generic "Error occurred..." to avoid expected-error log noise.

Annotated diff (important lines)
- return jd.doRefreshDSRangeListWithDB(l, tx.SqlTx())
+ if lock := jd.dsListLock.RTryLockWithCtx(ctx); !lock {
+   return fmt.Errorf("acquiring read lock ...: %w", ctx.Err()) // [1]
+ }
+ dsList, dsRangeList := jd.dsList.snapshot()                    // [2]
+ tx.setDSList(dsList, dsRangeList)
+ jd.dsListLock.RUnlock()

[1] Retry path now fails fast if lock acquisition cannot complete within caller context, instead of blocking on DB refresh.

[2] Assumes maintenance loops already published up-to-date dataset ranges into `jd.dsList`.

`jobsdb/jobsdb_test.go` (test alignment)

`TestUpdateJobStatusInExternalTxRetriesAfterReadonlyStatusTable` now explicitly repopulates in-memory DS list before invoking the stale retry. This is crucial because runtime retry semantics changed from DB refresh to cache snapshot.

`jobsdb/bench/scenario/compaction.go` (new benchmark scenario)

New 437-line benchmark harness runs three compaction strategy configurations against the same precomputed failure pattern and fixed dataset geometry. This makes comparisons fair and repeatable.

  • Supports dedicated maintenance DB pool (`JobsDB.Bench.maintenanceDSN`) to isolate maintenance SQL from consumer traffic.
  • Creates deterministic dataset rollover using `WithTriggerAddNewDS` and explicit trigger channel sends.
  • Uses global completion signal (`remaining == 0`) instead of treating empty `GetToProcess` response as drained, because `dsLimit` can hide pending jobs temporarily.
Scenario loop and flags
scenarios := []flagSet{

Cross-file Context Reviewers Should Know

  • `ErrStaleDsList` is emitted when status insert hits readonly-table trigger (`RS001`) in `doUpdateJobStatusInTx`; this is expected during compaction transitions (`jobsdb/jobsdb.go`, around line 3026 and `migration.go` comments around line 724).
  • Compaction code path (`doCompactDS`) explicitly states that status writers should treat trigger-induced errors as stale-list retries, and that DS list publication happens after commit via `doRefreshDSRangeList`.
  • `WithMaintenancePoolDB` already exists to isolate maintenance operations. The new benchmark leverages this to model pool contention while preventing benchmark artifacts from maintenance starvation.
  • `TriggerAddNewDS` existed as a field used in tests; this PR formalizes an options helper (`WithTriggerAddNewDS`) so benchmark setup can control rollover deterministically without manual struct wiring.
  • The PR description and benchmark motivation focus on jobsdb behavior, but actual branch also contains workflow changes. Review/approval policy may need to treat these as separate concerns even if both are valid.

How To Review This PR

  1. Start with `jobsdb/jobsdb.go` retry hunk in `UpdateJobStatusInTx`. Verify lock/refresh semantics are sound: lock acquired before snapshot, lock always released, no DB call in retry path.
  2. Jump to `jobsdb/migration.go` comments around `doCompactDS`. Confirm the assumption "compaction publishes cache" really holds for all stale-list cases, not only one migration variant.
  3. Check updated test in `jobsdb/jobsdb_test.go` to ensure it still validates external TX stale retry behavior and did not only mask a regression.
  4. Review benchmark files for representativeness and determinism. Ensure `RUN_COMPACTION_BENCH` gating and heavy resource settings are intentional and documented.
  5. Finally review `.github/workflows/ai-preview.yaml` as separate security/ops scope: sandbox bypass rationale, branch bootstrap logic, and symlink defense.

Specific things to scrutinize

  • Any path where stale-list retry could race with cache publication and still keep old DS range in `tx`.
  • Whether `RTryLockWithCtx` semantics could cause premature failure under lock contention where old behavior might have eventually succeeded.
  • Whether workflow cleanup (`stray` file removal) could remove intended content if `UMBRELLA_DIR` naming ever changes.

Risks, Edge Cases, Double-checks

HIGH Cache freshness is now a hard dependency. If any layout-changing path forgets to republish `jd.dsList`, stale retry may spin/fail against outdated ranges because it no longer self-heals via direct DB refresh.
MEDIUM Lock-acquire failure behavior changed. `RTryLockWithCtx` fails on context timeout/cancel; verify caller contexts are tuned so transient contention does not surface avoidable user-facing failures.
MEDIUM Mixed-scope PR increases review blind spots. JobsDB and CI workflow updates are unrelated enough that reviewers may under-review one side. Consider split PRs in future for cleaner ownership.
LOW Benchmark may be costly/noisy in shared CI if accidentally enabled. Gated by env var, but still worth ensuring this does not leak into default test paths.
LOW Workflow `xargs -d '\n'` behavior is GNU-specific. Safe on Ubuntu runners, but portability is narrower if workflow ever migrates to non-GNU environments.
