TL;DR

Label-gated GitHub Action builds an AI visual PR review site and publishes it to gh-pages, then removes it on PR close.
  • Trigger narrowed to pull_request: labeled, closed; rebuild requires re-adding ai-preview.
  • Uses GitHub App token, collects pr.diff/meta.txt, runs Codex, deploys pr-preview/pr-<N>/.
  • Adds safety checks: clean gh-pages init, stray-file cleanup, symlink rejection, sticky PR comment upsert/teardown.

Why

Review onboarding is slow for large PRs. This adds an opt-in preview workflow that auto-generates a visual explainer page and posts one stable URL in the PR, while keeping teardown and deployment behavior explicit.

Pain points map directly to label-driven build and close-driven teardown.

What Changed At A Glance

Two new files define trigger, generation, deployment, and reviewer-facing UX.
  • Job runs for non-fork PRs only, using a GitHub App token with scoped permissions.
  • Preview build happens only on ai-preview label application, not on pushes.
  • Codex runs from repo root and must produce review-site/index.html.
  • Deployment isolates content under pr-preview/pr-<N>/ plus .nojekyll.
  • Close event removes preview directory and rewrites sticky comment state.

Risks

High

Sandbox bypass allows unrestricted Codex process inside runner job.

Mitigation: keep label restricted and app/secrets tightly scoped.

Medium

Preview does not refresh on new commits unless label is re-added.

Mitigation: document refresh behavior in team PR workflow.

Medium

Cleanup runs only on close; stale preview may remain for long-lived open PRs.

Mitigation: close superseded PRs or add manual cleanup command later.

Low

Prompt is strict; model output may still vary in quality and completeness.

Mitigation: fail fast on missing file; review generated page before trust.
