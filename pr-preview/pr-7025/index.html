TL;DR

Adds an opt-in GitHub Actions pipeline that generates a multi-stage AI PR review site and deploys/removes per-PR previews on GitHub Pages.

  • Introduces a four-stage Codex pipeline using `review-work/` artifacts (`analysis`, `diagrams`, `risks`, `site`).
  • Hardens preview publishing with GitHub App auth, safe metadata collection, orphan gh-pages setup, and symlink rejection.
  • Changes trigger behavior so build/deploy happens on `ai-preview` labeling and teardown happens on PR close only.

Why

Reviewing unfamiliar PRs is slow without context. This workflow creates a visual, structured onboarding page from the diff and repository, then publishes it behind an explicit label with automated cleanup on PR closure.

What Changed

  • Defines end-to-end trigger, context capture, Codex stages, gh-pages deploy, sticky comment upsert, and close-time teardown behavior.
  • Replaces monolithic prompt with strict orchestration that runs four stage-specific prompts and validates each produced artifact.
  • Introduces common rules for all stages, including grounding claims in `pr.diff`/repo and restricting file writes to owned outputs.
  • Specifies schema-rich `analysis.json` production with repository exploration, architecture components, flows, data shapes, and key hunks.
  • Defines deterministic SVG generation and manifesting for components, flows, and before/after visuals tied to analysis artifacts.

Risks

medium

On `pull_request` `synchronize` events (new commits), the preview is not regenerated because the workflow now only triggers on `labeled` and `closed`, so the published page can become stale while appearing current.

Require removing/re-adding the `ai-preview` label after each significant push (or restore `synchronize` trigger if always-fresh previews are required).

medium

`Collect PR context` falls back to `git diff "origin/${BASE_REF}...HEAD"`, but there is no explicit fetch of `origin/${BASE_REF}` beyond the checked-out head ref, so fallback can fail and break preview generation.

Fetch the base ref explicitly before fallback (for example `git fetch origin ${BASE_REF}`) or make `gh pr diff` failure non-fatal with a clearer diagnostic path.

low

The same workflow performs both deploy and teardown under `concurrency.cancel-in-progress: true`, so rapid label/close actions can cancel an in-flight job and leave preview/comment state temporarily inconsistent until a later event reruns.

Split deploy and teardown into separate jobs/concurrency groups or make cleanup idempotence checks robust enough to tolerate canceled intermediate runs.

Review Checklist

  • Check `.github/workflows/ai-preview.yaml` behavior on a PR commit push after labeling and confirm reviewers understand preview is intentionally stale until label reapply.
  • Check fallback robustness in `.github/workflows/ai-preview.yaml` by forcing `gh pr diff` failure and verifying whether `origin/${BASE_REF}` exists in the runner before `git diff` runs.
  • Check teardown path in `.github/workflows/ai-preview.yaml` removes both `pr-preview/pr-<N>/` and the sticky marker comment on close/merge for previously deployed previews.
  • Check `review-work/` stage contracts in `.github/codex/agents/*.md` enforce strict single-file outputs so one stage cannot overwrite artifacts owned by another stage.
