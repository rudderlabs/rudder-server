TL;DR

An opt-in ai-preview workflow now generates, deploys, comments, and tears down PR review sites on close.

  • Build runs only on labeled (ai-preview) and closed events.
  • Uses GitHub App token with scoped write permissions, not default workflow token.
  • Deploy path is isolated to gh-pages/pr-preview/pr-<N>/ with symlink rejection.

Why

Reviewers need fast PR context without reading full diffs. This adds an explicit, label-gated path to generate visual review pages and publish them safely, while cleaning up previews only when the PR closes.

Preview creation is explicit; cleanup is close-only.

What Changed At A Glance

Two new files drive the full preview lifecycle.
  • Workflow trigger narrowed to pull_request: [labeled, closed].
  • Build path runs only on action == labeled and label exactly ai-preview.
  • Teardown path always runs on closed, independent of labels.
  • Deployment hardening: clean gh-pages init, stray-file cleanup, symlink reject gate.
  • Prompt file defines strict visual-first output contract for generated review pages.
Changes are concentrated in automation and prompt instructions.
