TL;DR

Adds a label-gated GitHub workflow that runs four Codex stages to generate and deploy a single-file PR review dashboard to GitHub Pages.

Introduces `.github/workflows/ai-preview.yaml` to orchestrate generate/validate/deploy/teardown preview lifecycle.
Adds stage prompts under `.github/codex/agents/` to enforce a strict analyze→diagrams→risks→assemble pipeline via `review-work/` artifacts.
Adds CI checks and docs updates to keep generated HTML self-contained, diagram-backed, and safe for deployment.

Why

Reviewing unfamiliar PRs is slow; this PR automates a visual onboarding page so reviewers can quickly understand context, flow, risks, and review focus before deep diff analysis.

What Changed At A Glance

  • GitHub Actions preview orchestrator — Adds a label-gated workflow to run Codex stages, validate outputs, deploy previews, and tear them down when PR closes. .github/workflows/ai-preview.yaml
  • Codex multi-stage prompt suite — Defines shared conventions and four stage-specific prompts with strict input/output contracts. .github/codex/agents
  • Review site template and validation — Refines output rules and asset expectations for embedded diagrams and review-work references. review-site/index.html
  • Contributor documentation — Documents required secrets/variables, label usage, and stage-by-stage architecture for preview generation. README.md

Risks

high

The workflow installs and executes an unpinned `@openai/codex` from npm, so a compromised/new upstream release could execute arbitrary code with repo write and secrets access during CI.

Pin the Codex CLI to an exact version (or digest-verified artifact) and gate upgrades through reviewed PRs.

medium

The workflow writes the full PR title/body and commit subjects into `meta.txt` and sends them to the model, which can exfiltrate sensitive text included in PR content to the external LLM endpoint.

Add explicit redaction/allowlist filtering for secrets and sensitive patterns before building `meta.txt`, or require opt-in labels only on trusted internal PRs.

medium

Preview cleanup only runs on `pull_request.closed`, so stale `gh-pages/pr-preview/pr-<n>` directories persist if labeling strategy changes or teardown fails once, leaving outdated reviewer artifacts publicly served.

Add periodic reconciliation (scheduled cleanup) that removes preview directories for non-open PRs and validate teardown success with alerting.

low

The documented trigger semantics changed to label-only regeneration; new commits after initial labeling are not reflected unless label is re-applied, so reviewers can trust stale previews.

Surface a prominent staleness indicator (head SHA/timestamp) in the generated page and optionally support `synchronize` events for automatic refresh.

Review Checklist

  • Verify `.github/workflows/ai-preview.yaml` pins every executable action/tool version, including `@openai/codex`, before enabling in protected repos.
  • Check that `Collect PR context` in `.github/workflows/ai-preview.yaml` cannot include sensitive data in `meta.txt` that should not leave GitHub/Azure boundaries.
  • Confirm teardown path in `.github/workflows/ai-preview.yaml` deletes `pr-preview/pr-<number>` on close and that a failed push cannot silently leave stale pages.
  • Check generated `review-site/index.html` includes PR head SHA and generation time so humans can detect stale previews after new commits.
  • Verify GitHub App permissions (`contents:write`, `pull-requests:write`) used in `.github/workflows/ai-preview.yaml` are the minimum required for deploy + comment updates.
