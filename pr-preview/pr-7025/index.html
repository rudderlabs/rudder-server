TL;DR

Adds a label-gated GitHub Actions pipeline that uses four Codex stages to generate, validate, deploy, and tear down a PR review dashboard on GitHub Pages.

  • 💡Introduces a four-stage artifact handoff model using `review-work/` between analyze, diagrams, risks, and assemble prompts.
  • 📌Adds `.github/workflows/ai-preview.yaml` with label gating, concurrency cancellation, deploy safeguards, and sticky PR preview comments.
  • 🧩Hardens pipeline checks by validating diagram manifests, embedding constraints, directory setup, and safe `gh-pages` lifecycle handling.

Why

Reviewing unfamiliar PRs is slow and context-heavy; this adds an opt-in, visual-first preview that summarizes architecture, flow, risks, and review steps in one deployed page.

Risks

medium

Preview updates no longer happen on new commits because the workflow now only triggers on labeled/closed events, so deployed review pages can become stale while a PR evolves.

Either reintroduce `synchronize`/`reopened` triggers or clearly enforce and document mandatory relabeling after every push.

medium

Each run globally installs the latest `@openai/codex` without version pinning, so behavior and output contracts can change unexpectedly and break artifact validation or HTML assembly.

Pin `@openai/codex` to a known version (or lock by checksum) and update intentionally with a compatibility check.

low

The workflow writes app-token-authenticated remote URLs into `.git/config` in temporary clones and does not scrub them, increasing accidental credential exposure risk if later debug/log steps ever print git config.

Prefer `actions/checkout` with token inputs or explicitly reset/remove remotes after push/cleanup steps.

Review checklist

  • Check `.github/workflows/ai-preview.yaml` trigger behavior matches intended lifecycle: relabel-only regeneration is explicitly acceptable for reviewers.
  • Check `.github/workflows/ai-preview.yaml` cleanup path reliably deletes `pr-preview/pr-<number>` and updates/removes sticky comments on close events.
  • Check `.github/workflows/ai-preview.yaml` label gates are consistent (`ai-preview` in job `if` and comment/deploy logic) so no mismatched label leaves orphaned previews.
  • Check `.github/codex/agents/assemble.md` constraints (`data-diagram` ids, missing SVG warnings, required sections) still tolerate real outputs from analyze/diagrams stages.
  • Check `review-work` artifact schema compatibility end-to-end: `analysis.json`, `diagrams.json`, and `risks.json` fields consumed by the assembler are all produced by upstream stage prompts.
