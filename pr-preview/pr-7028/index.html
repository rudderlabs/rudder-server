TL;DR

Stale-dataset retries stop querying DB and now reuse in-memory DS snapshots, while a new heavy compaction benchmark stresses the exact contention path.

  • 🧠 Retry path changed in UpdateJobStatusInTx: read-lock + snapshot, no refresh query.
  • 🧪 Added compaction benchmark scenario (2M jobs shape in test config).
  • 🔧 New hook WithTriggerAddNewDS enables deterministic dataset rollover in tests/bench.

Why

Under high updater concurrency, stale-DS retries could deadlock on main DB pool exhaustion because retry needed a DB-backed DS refresh while all connections were busy.

Stale retry removes DB dependency from the hot update loop.

What Changed At A Glance

Most churn is the new compaction benchmark scenario.
1UpdateJobStatusInTx now uses RTryLockWithCtx and in-memory dsList.snapshot() on stale retries.
2internalUpdateJobStatusInTx suppresses generic error log for ErrStaleDsList to reduce noise.
3WithTriggerAddNewDS option added; benchmark injects deterministic dataset rollover trigger.
4New bench scenario compares three compaction flag sets under identical precomputed failure pattern.
5Test updated to refresh in-memory DS list before retry, matching the new stale-retry contract.
