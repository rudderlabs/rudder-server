TL;DR

Stale-dataset retries in status updates now reuse in-memory DS snapshots instead of DB refreshes, and a heavy compaction benchmark was added to measure draining behavior.

Retry path removes one DB dependency New compaction scenario compares 3 flag sets Lock-duration logging added in migration paths

Why

When compaction changes dataset layout mid-update, status writes can hit ErrStaleDsList. The old retry fetched DS metadata from DB, which can deadlock under pool saturation. The fix snapshots the already-refreshed in-memory DS list behind a read lock.

Retry path swaps DB refresh for lock-guarded in-memory snapshot.

What Changed At a Glance

Most churn sits in the new compaction benchmark scenario.
  • UpdateJobStatusInTx: on stale DS, rollback to savepoint, read-lock, snapshot DS list, retry.
  • internalUpdateJobStatusInTx: suppresses generic error log noise for expected stale retries.
  • New option WithTriggerAddNewDS enables deterministic dataset rollover in benches/tests.
  • New compaction bench scenario compares baseline, non-blocking drop, and non-blocking compaction.
  • Tests now manually refresh in-memory DS list before retry assertion in stale-table replacement case.
